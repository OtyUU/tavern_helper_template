# Plan: `{{player_status}}` Macro for LLM Prompt

## Goal

Inject a readable scene-status summary into the LLM prompt via a `{{player_status}}` macro. The macro must behave differently depending on generation type:

- **New message / continue / impersonate**: compute status AFTER the last message in chat.
- **Swipe / regenerate**: compute status AFTER the message BEFORE the one being regenerated (skip the message being replaced).

This matters because a swipe is a re-generation of the same message position — the current content of that message is stale/wrong, so the LLM should see the scene state from *before* it.

## Approach

Use `registerMacroLike()` to register a macro that resolves `{{player_status}}` at prompt-build time.

To distinguish generation types, listen to `tavern_events.GENERATION_STARTED` which fires **before** prompt assembly and receives a `type` parameter (`'normal'`, `'swipe'`, `'regenerate'`, `'continue'`, etc.). Store the generation type in a module-level variable that the macro callback reads.

Event ordering in SillyTavern:
1. `GENERATION_STARTED(type, option, dry_run)` ← capture type here
2. Prompt assembly begins ← macros expanded here
3. API request sent
4. `GENERATION_ENDED(message_id)` ← reset type here

## Files to Create

### `src/renpy-player/status-macro.ts` (new)

#### Module state

```ts
let activeGenerationType: string | null = null;
```

#### `formatPlayerStatus(state: InitialPlayerState): string`

Formats an `InitialPlayerState` into a human-readable block:

```
[Scene Status]
Background: living_room (night)
Camera: closeup
Characters:
- chinami: outfit=pajamas pose=base expression=neutral position=left
- eileen: outfit=uniform pose=stand expression=smile position=right blush
```

- If no background is set, omit the Background line.
- If camera is default (undefined), omit the Camera line.
- If no visible sprites, show "Characters: none".
- If state is completely empty (no background, no sprites), return an empty string.

#### `computePlayerStatus(): string`

Wrapped in try-catch — any error returns empty string so a broken macro never blocks generation:

```ts
export function computePlayerStatus(): string {
  try {
    return computePlayerStatusInner();
  } catch (error) {
    console.error('[RenPy Player] Error computing player_status:', error);
    return '';
  }
}
```

**`computePlayerStatusInner()` (private):**

1. Read settings from the Pinia store (`useRenpyPlayerSettingsStore`).
2. Determine how far to replay based on generation type:
   ```ts
   const lastId = getLastMessageId();
   if (lastId < 0) return '';

   if (activeGenerationType === null) {
     console.warn('[RenPy Player] player_status: generation type unknown, defaulting to full history');
   }

   const isRegen = activeGenerationType === 'swipe'
                || activeGenerationType === 'regenerate';
   const upToId = isRegen ? lastId - 1 : lastId;
   if (upToId < 0) return '';
   ```
3. Collect messages from `upToId` down to `0` in **reverse order** (newest first), as `getInitialState` expects:
   ```ts
   const messagesBackwards: string[] = [];
   for (let id = upToId; id >= 0; id--) {
     const msg = getChatMessages(id)[0];
     if (msg?.message) {
       messagesBackwards.push(msg.message);
     }
   }
   ```
4. Build `FrameBuildOptions` from settings (empty `assetRoot` and `assetExtensions` — no asset resolution needed).
5. Call `getInitialState(messagesBackwards, buildOptions)`.
6. Return `formatPlayerStatus(state)`.

#### `registerPlayerStatusMacro(): { unregister: () => void }`

```ts
export function registerPlayerStatusMacro(): { unregister: () => void } {
  const startedHandle = eventOn(tavern_events.GENERATION_STARTED, (type) => {
    activeGenerationType = type;
  });
  const endedHandle = eventOn(tavern_events.GENERATION_ENDED, () => {
    activeGenerationType = null;
  });
  const macroHandle = registerMacroLike(
    /\{\{player_status\}\}/gi,
    () => computePlayerStatus(),
  );

  return {
    unregister: () => {
      startedHandle.stop();
      endedHandle.stop();
      macroHandle.unregister();
      activeGenerationType = null;
    },
  };
}
```

## Generation Type Semantics

| Generation type | What's happening | upToId | Rationale |
|---|---|---|---|
| `'normal'` | New message after last | `lastId` | Scene after all existing messages |
| `'continue'` | Appending to last message | `lastId` | Scene after the message being continued |
| `'swipe'` | Replacing last message | `lastId - 1` | Skip the stale message being replaced |
| `'regenerate'` | Same as swipe | `lastId - 1` | Same rationale |
| `'impersonate'` | Generating user message | `lastId` | Scene after all existing messages |
| `'quiet'` | Silent generation | `lastId` | Default: include all |
| `'first_message'` | First message of chat | `-1` → empty | No messages yet, no scene |

## Edge Cases

- **No chat messages / empty chat**: `getLastMessageId()` returns -1. Return empty string.
- **Regenerating first message**: `upToId = lastId - 1 = -1`. Return empty string.
- **Messages with no script commands**: `parseScriptFromMessage` returns `source: 'none'`, commands empty. State inherited from earlier messages or empty.
- **Macro used outside generation** (e.g., in a world info entry during display): `activeGenerationType` is `null`, which is NOT `'swipe'`/`'regenerate'`, so `upToId = lastId` (include all). This is the safe default.
- **Performance**: `getInitialState` replays all messages. For long chats this could be slow. Acceptable for once-per-generation. Can cache later if needed.

## Dependencies on Settings

The status computation needs these settings from the Pinia store:
- `characterSpriteConfig` — per-character pose tokens and default outfits
- `defaultPose`, `defaultExpression` — fallbacks
- `globalPoseTokens` — global pose token list

Asset-related settings (`assetRoot`, `assetExtensions`) are passed as empty since we don't resolve file paths for status text.
