# Plan: Replace O(N) `getChatMessages` loops with single range query

## Problem

Both `useRenpyPlayerController.ts` (lines 108–112) and `status-macro.ts` (lines 72–77) build chat history by calling `getChatMessages(id)` individually for every message ID in a loop. With 200 messages, this fires 200 separate API calls every time `frames` recomputes (every chat event).

## Root Cause

The Tavern Helper `getChatMessages` API supports range queries like `getChatMessages('0-10')` that return all messages in one call, sorted by `message_id`. This capability is not being used.

## How `getInitialState` expects input

`getInitialState(messagesBackwards, options)` iterates from `messagesBackwards.length - 1` down to `0` (parser.ts:735). This means:
- `messagesBackwards[0]` = **newest** message
- `messagesBackwards[last]` = **oldest** message

The function processes oldest → newest, which is the correct replay order.

`getChatMessages('0-N')` returns messages sorted by `message_id` **ascending** (oldest first). So the result must be **reversed** to match the `messagesBackwards` convention.

## Changes

### 1. `useRenpyPlayerController.ts` (lines 105–114)

Replace the per-ID loop with a single range query:

```ts
// BEFORE:
const history: string[] = [];
const currentId = currentMessage.value?.message_id;
if (currentId != null && !Number.isNaN(currentId)) {
  for (let id = currentId - 1; id >= 0; id -= 1) {
    const msg = getChatMessages(id)[0];
    if (msg && msg.message) {
      history.push(msg.message);
    }
  }
}

// AFTER:
const history: string[] = [];
const currentId = currentMessage.value?.message_id;
if (currentId != null && !Number.isNaN(currentId) && currentId > 0) {
  const allMessages = getChatMessages(`0-${currentId - 1}`);
  for (let i = allMessages.length - 1; i >= 0; i--) {
    if (allMessages[i]?.message) {
      history.push(allMessages[i].message);
    }
  }
}
```

Notes:
- Added `currentId > 0` guard — when `currentId` is 0, there's no history to fetch. The original loop implicitly handled this (started at `-1`, never entered the loop). The explicit guard is clearer.
- Iterates `allMessages` backwards (from last index to 0) to build the `history` array in newest-first order without allocating an intermediate reversed array.
- Skips messages with falsy `.message` (same filter behavior as original).

### 2. `status-macro.ts` (lines 71–77)

Replace the per-ID loop with a single range query:

```ts
// BEFORE:
const messagesBackwards: string[] = [];
for (let id = upToId; id >= 0; id--) {
  const msg = getChatMessages(id)[0];
  if (msg?.message) {
    messagesBackwards.push(msg.message);
  }
}

// AFTER:
const allMessages = getChatMessages(`0-${upToId}`);
const messagesBackwards: string[] = [];
for (let i = allMessages.length - 1; i >= 0; i--) {
  if (allMessages[i]?.message) {
    messagesBackwards.push(allMessages[i].message);
  }
}
```

Notes:
- `upToId >= 0` is already guaranteed at this point (line 69 returns early otherwise).
- Uses the same reverse-iteration pattern as the controller fix.

## Edge Cases

| Scenario | Original behavior | New behavior |
|---|---|---|
| `currentId = 0` (controller) | Loop starts at `-1`, never enters → empty history | Guarded by `currentId > 0` → empty history |
| Deleted messages (gaps in IDs) | `getChatMessages(deletedId)[0]` returns `undefined`, skipped | Range query returns only existing messages, no gaps to skip |
| `upToId = 0` (status-macro) | Single call for message 0 | `getChatMessages('0-0')` — returns array with 0 or 1 elements |

## Verification

- The `history` / `messagesBackwards` arrays are only consumed by `getInitialState()`, which is not changing.
- Both call sites produce identical arrays (same content, same order) — the refactoring is purely about how the API is called.
- No changes to `parser.ts`, `types.ts`, or any other file.
