# Plan: Stable Render Resolution (Fixed renderScale Inner Stage)

## Overview
Decouple render resolution from zoom. Introduce a fixed `renderScale` that determines the "inner stage" size. Camera zoom becomes a downscale of the inner stage via CSS `transform: scale()`. Camera layers become translate-only. Background gets an independent extra-scale wrapper for separate bg vs sprite zoom.

## Files to modify

### 1. `src/renpy-player/useRenpyPlayerController.ts`

**Add computed properties** (after `stageWidth` / `hudScale` ~line 713):

```ts

const renderScale = computed(() => (dpr >= 2 ? 1.5 : 2));

const innerStageWidth = computed(() => Math.round(stageWidth.value * renderScale.value));
const innerStageHeight = computed(() => Math.round(settings.value.stageHeight * renderScale.value));

const innerStageScale = computed(() => spriteZoom.value / renderScale.value);

const bgExtraScale = computed(() => {
  if (spriteZoom.value === 0) return backgroundZoom.value;

```

Wait, let me reconsider `bgExtraScale`. The inner stage is scaled by `innerStageScale = spriteZoom / renderScale`. The background should appear at `backgroundZoom` visual scale relative to the original stage. Inside the inner stage (which is `renderScale` times bigger), the bg layer has translate-only. To achieve visual `backgroundZoom` from outside:

- Visual bg zoom = `innerStageScale * bgExtraScale` should equal `backgroundZoom / renderScale * renderScale / renderScale`... let me think more carefully.

The inner stage has visual size `innerStageWidth * innerStageScale`. The inner stage width is `stageWidth * renderScale`. The inner stage scale is `spriteZoom / renderScale`. So visual inner stage width = `stageWidth * renderScale * spriteZoom / renderScale` = `stageWidth * spriteZoom`. Good, that gives visual sprite zoom.

For background: we want visual bg zoom = `backgroundZoom`. Currently inside the inner stage, bg is at 1:1 (translate only). So visual bg = `innerStageScale * 1` = `spriteZoom / renderScale`. We need extra bg scale so `innerStageScale * bgExtraScale = backgroundZoom`:
- `bgExtraScale = backgroundZoom / innerStageScale = backgroundZoom * renderScale / spriteZoom`

But wait, the background zoom already includes parallax. So:

```ts
const bgExtraScale = computed(() => {
  return spriteZoom.value > 0 ? backgroundZoom.value / spriteZoom.value : 1;
});
```

Hmm, but that doesn't account for renderScale. Let me re-derive:

- Outside the inner stage, everything is scaled by `innerStageScale = spriteZoom / renderScale`
- Inside the inner stage, sprites are at 1:1 pixel mapping (renderScale resolution)
- So sprites appear at `spriteZoom / renderScale * renderScale = spriteZoom` visual zoom. Correct.
- For background, inside the inner stage, we add extra scale `bgExtraScale`
- Visual bg zoom = `innerStageScale * bgExtraScale = (spriteZoom / renderScale) * bgExtraScale`
- We want this to equal `backgroundZoom`
- So `bgExtraScale = backgroundZoom * renderScale / spriteZoom`

Actually wait. The background inside the inner stage is already at `renderScale` resolution. The SmartImage fills the inner stage dimensions (`innerStageWidth x innerStageHeight`), which is `stageWidth * renderScale` by `stageHeight * renderScale`. So the background image fills a `renderScale`-larger area at 1:1 inside the inner stage.

When the inner stage is scaled by `innerStageScale = spriteZoom / renderScale`:
- The visual background size becomes `stageWidth * renderScale * (spriteZoom / renderScale)` = `stageWidth * spriteZoom`
- This gives us `spriteZoom` visual zoom on the background, same as sprites.

To get `backgroundZoom` instead, we need `bgExtraScale` inside the inner stage:
- Visual bg = `stageWidth * renderScale * bgExtraScale * (spriteZoom / renderScale)` = `stageWidth * spriteZoom * bgExtraScale`
- We want this to be `stageWidth * backgroundZoom`
- So `bgExtraScale = backgroundZoom / spriteZoom`

```ts
const bgExtraScale = computed(() =>
  spriteZoom.value > 0 ? backgroundZoom.value / spriteZoom.value : 1
);
```
