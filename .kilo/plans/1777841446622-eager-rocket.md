# Fix: `startReveal` double-trigger on frame recomputation

## Problem

`watch(currentFrame, ...)` in `useDialogueReveal` (`player-composables.ts:551`) fires on **reference** changes. When `frames` is recomputed (e.g. via `fullSync` → `historyTrigger++` → `inheritedState` → `frames`), `currentFrame` gets a new object even when the logical content is identical. This causes `startReveal` to re-fire, resetting `revealedCharCount = 0` and restarting the typing animation from scratch.

## Root Cause

`currentFrame = computed(() => frames.value[frameIndex.value] ?? null)` — `buildFrames()` returns new objects each call, so the watcher sees a different reference.

## Fix

Replace the reference-based watcher with a primitive-based one using `currentFrame.value?.index` (a stable number).

### File: `src/renpy-player/player-composables.ts`

Replace lines 551–567:

```ts
// BEFORE
watch(
  currentFrame,
  (nextFrame, prevFrame) => {
    if (!nextFrame || !nextFrame.text) {
      revealGeneration++;
      graphemes.value = [];
      revealedCharCount.value = 0;
      displayedSpeaker.value = '';
      speakerRevealed.value = true;
      isRevealing.value = false;
      isFullyRevealed.value = true;
      return;
    }
    startReveal(nextFrame.text, prevFrame?.speaker, nextFrame.speaker);
  },
  { immediate: true },
);
```

With:

```ts
// AFTER
let trackedPrevFrame: PlayerFrame | null = null;

watch(
  () => currentFrame.value?.index ?? -1,
  () => {
    const nextFrame = currentFrame.value;
    const prevFrame = trackedPrevFrame;
    trackedPrevFrame = nextFrame ?? null;

    if (!nextFrame || !nextFrame.text) {
      revealGeneration++;
      graphemes.value = [];
      revealedCharCount.value = 0;
      displayedSpeaker.value = '';
      speakerRevealed.value = true;
      isRevealing.value = false;
      isFullyRevealed.value = true;
      return;
    }
    startReveal(nextFrame.text, prevFrame?.speaker, nextFrame.speaker);
  },
  { immediate: true },
);
```

### Why this works

- `index` is a primitive number set by `buildFrames` based on array position. When `frames` is recomputed with identical content, `frames[1].index` is still `1` — the watcher does not fire.
- On a real frame transition (`frameIndex` changes), the index changes, so the watcher fires correctly.
- `trackedPrevFrame` is maintained locally to track the previous frame's speaker for speaker transition logic. On the first invocation (`immediate: true`), `prevFrame` is `null`, which is correct.
